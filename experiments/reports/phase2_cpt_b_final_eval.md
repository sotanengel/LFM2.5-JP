# Phase 2 cpt-B 完走モデル最終再検証(Stage V)

実施日: 2026-07-11 / 環境: WSL2 Ubuntu + RTX 3060 Ti(GPU 1枚・全ジョブ直列実行)

対象モデル: `outputs/cpt-1.2b-layerft`(層FT・LiquidAI/LFM2.5-1.2B-Base を起点に CPT、
checkpoint-1000〜12000(1000刻み)+ checkpoint-12941 = 最終出力ルート)

## 評価条件

- **凍結条件**: 3 種の評価はいずれも Stage 0 で実装済みのコードをそのまま実行(コード変更なし)。
  llm-jp-eval 推論側の venv(`~/llm-jp-eval-inference/inference-modules/transformers`)は
  accelerate 更新 + numpy<2 固定 + `inference.py` の `generation_config` 行削除という手パッチ済み
  状態を凍結して使用(`uv sync` は一切実行せず `uv run --no-sync` のみ)。実行前に
  numpy=1.26.4 / accelerate=1.14.0 が維持されていることを確認済み。
- **held-out 定義**: `configs/data/corpus.yaml` の wikipedia_ja から `--skip 100000 --take 200`
  で抽出した 200 文書(`data/processed/heldout_ja.jsonl`)。学習データの packing 範囲と重ならない
  よう先頭 100,000 文書をスキップしてサンプリング。
- **日本知識プローブ v2 の採点方式**: 10 分野 × 5 問、1-shot・greedy decoding。
  `extract_answer_segment()` で生成テキストから先頭の答えセグメントを切り出し、想定解のサブ
  ストリング一致で正誤判定(Issue #76: 選択肢を列挙する解答から偶然正解語を拾ってしまう旧採点の
  バグを修正した版)。
- **既知の実行時課題**: `quick_eval.py` の `ppl` サブコマンドはトークナイザを評価対象パス自身から
  ロードする実装だが、CPT checkpoint ディレクトリ(checkpoint-5000/9000/12000)および最終出力
  ルート `outputs/cpt-1.2b-layerft/` にはモデル重みのみが保存されておりトークナイザファイル
  (tokenizer.json 等)が存在しないため初回実行は失敗した。コードは変更せず、HF キャッシュ上の
  base モデルスナップショットから `tokenizer.json` / `tokenizer_config.json` /
  `special_tokens_map.json` / `chat_template.jinja` を該当ディレクトリへコピーする形で回避した
  (トークナイザは CPT で変化しないため妥当な回避策。`japan_probe.py` は元々 base モデル定数から
  常にトークナイザをロードする設計のためこの問題の影響を受けていない)。

**旧採点(v1)との差分(過大評価の定量化)**: 旧採点では base 25 / ckpt5000 25 / ckpt9000 **28** だった。
新採点(v2, Issue #76 修正版)では base 25 / ckpt5000 25 / ckpt9000 **25** となり、ckpt9000 は
**3 点分過大評価**されていたことが判明した。原因は「琵琶湖がある都道府県」「大政奉還を行った将軍」
のように選択肢を列挙する形の解答で、旧ロジックが解答本文以外の箇所から偶然正解語を拾って正答扱い
していたケース(v2 レポート内の「旧判定との差分」列に ckpt9000 の該当 2 件を記録)。final は
v1 では未計測(v2 で新規計測、26/50)。

修正後の傾向としては、素朴な日本語知識 QA では base→final で **ほぼ横ばい〜わずかに改善**
(25→25→25→26)にとどまり、旧採点が示していたような「ckpt9000 で明確に伸びる」という傾向は
再現しなかった。

## Phase 2 ゲート観点の事実整理(判定はオーケストレータに委ねる)

1. **評価軸によって cpt-B の評価が逆転する**。held-out ppl(表1)では CPT の進行とともに一貫して
   改善し(9.12→8.36)、日本知識プローブ v2(表2)でも base→final でわずかに改善(25→26)。
   一方 llm-jp-eval 凍結条件(表3)では、CPT 前の base(AVG 0.2347)より CPT 後の
   ckpt9000/final(AVG 0.2127 / 0.2160)の方が**低い**。特に jsick・jnli・niilc が base から
   一様に低下し、jsts の pearson 相関は base でほぼ 0 だったものが CPT 後には**負の相関**
   (-0.15 / -0.06)に転じている。
2. **cpt-B は cpt-C に一貫して劣る**。表3の全指標で cpt-B(base/9000/final いずれも AVG
   0.21〜0.23)は cpt-C(JP-202606, AVG 0.3867)を大きく下回り、素の Instruct(AVG 0.334)にも
   届いていない。jcommonsenseqa の exact_match は cpt-B が 0.15〜0.22 に対し cpt-C は 0.78 と
   差が大きい。
3. **旧採点の過大評価が一部で解消された**(表2)ことで、「ckpt9000 が知識面で明確に強い」という
   従来の見立ては再現しなくなった。ただし held-out ppl は素直に改善しているため、CPT がドメイン
   適応(wikipedia_ja に対する言語モデリング)には効いている一方、llm-jp-eval が要求する few-shot
   構造化出力(選択肢からの厳密一致解答、JSON的な出力形式追従など)への適合や、素朴な知識想起の
   両方では明確な改善を示せていない、という乖離が観測される。
4. 上記はいずれも「事実」の整理であり、Phase 2 ゲートの通過・不通過判定そのものは行わない
   (オーケストレータの判断に委ねる)。
